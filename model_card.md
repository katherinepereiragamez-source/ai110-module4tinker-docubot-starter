DocuBot Model Card
1. System Overview
What is DocuBot trying to do?

DocuBot is a question-answering assistant grounded in a specific set of project documentation files. Its goal is to answer developer questions accurately using only the evidence in those files, and to refuse clearly when that evidence is absent. It is not a general-purpose assistant — staying inside the docs is a feature, not a limitation.

What inputs does DocuBot take?

A natural-language question from the user (string)
A folder of .md and .txt documentation files loaded at startup
An optional GeminiClient instance (required only for RAG and naive LLM modes)
What outputs does DocuBot produce?

Retrieval only: raw paragraph-level chunks from the most relevant docs, labelled with source filenames
RAG: a synthesised prose answer generated by the LLM, grounded in those same chunks
Naive LLM: a generated answer using the entire corpus as context, with no retrieval filter applied
Refusal: the message "I don't have enough information in these docs to answer that." when no chunk clears the confidence threshold
2. Retrieval Design
How does your retrieval system work?

Indexing: At startup, every document is split into paragraphs (on \n\n boundaries). Each paragraph shorter than 40 characters is discarded as noise. The remaining chunks are indexed in an inverted index that maps each lowercase token to the set of chunk positions containing it. This runs once and is reused on every query.
Scoring: For a given query, candidate chunks are identified via the index (only chunks containing at least one query word are considered). Each candidate is scored by counting how many unique query words appear anywhere in the chunk. Repeated words in the query don't inflate the score — "token token token" is treated the same as "token".
Selection: Candidates are sorted by score descending, with filename as a stable alphabetical tiebreaker. Only chunks scoring ≥ MIN_SCORE_THRESHOLD (currently 2) are returned. The top 3 are passed to the answer layer.
What tradeoffs did you make?

Decision	Benefit	Cost
Paragraph chunks over whole documents	More precise, less noise for the LLM	A single-sentence chunk might lack surrounding context
Term overlap scoring over TF-IDF	Simple to reason about and debug	Common words like "the" or "error" score the same as rare ones
MIN_SCORE_THRESHOLD = 2	Reduces low-confidence guesses	Single-word queries always refuse, even when docs are relevant
Index built at startup	Fast queries	Docs must be present at launch; live updates aren't supported
3. Use of the LLM (Gemini)
When does DocuBot call the LLM and when does it not?

Naive LLM mode: The LLM is called with the entire corpus concatenated into the context window. No retrieval happens. The model answers from whatever it can find in that wall of text, plus its own training knowledge. This is the least controlled mode.
Retrieval only mode: The LLM is never called. The system returns raw chunk text directly, labelled by source file. Fast, transparent, and auditable, but requires the user to interpret the evidence themselves.
RAG mode: Retrieval runs first. Only the top-scoring chunks are passed to the LLM. The LLM is instructed to use only those chunks and to say "I don't know" if they don't contain an answer.
What instructions do you give the LLM to keep it grounded?

The system prompt for RAG mode enforces these rules:

Answer using only the provided snippets — do not use outside knowledge
If the snippets do not contain enough information, respond with "I don't know based on the provided documentation"
Cite the source filename for any claim you make
Do not speculate, infer, or extrapolate beyond what is explicitly stated in the snippets

What patterns did you notice?

When naive LLM looks confident but is weakly grounded: When asked about specific implementation details (endpoint paths, parameter names, env variable names), the LLM tends to produce plausible-sounding answers drawn from its training on similar codebases. The output reads fluently and gives no signal that it may be fabricated. This is the most dangerous failure mode — wrong answers that feel trustworthy.
When retrieval only is clearly better: For questions with a precise, locatable answer ("which env variable holds the DB password?"), retrieval only returns the exact paragraph. No interpretation needed, no risk of the LLM paraphrasing something incorrectly. The cost is that the user has to read and extract the answer themselves.
When RAG is clearly better than both: For questions that require a short, synthesised explanation rather than a raw chunk ("how does a client refresh a token?"), RAG produces a readable answer grounded in real evidence. It combines the accuracy of retrieval with the usability of generation.
When RAG still fails: If the relevant information is split across two paragraphs that score below the threshold individually, neither chunk is retrieved and RAG refuses to answer — even though the answer exists in the docs. This is a retrieval failure, not an LLM failure.
5. Failure Cases and Guardrails
Failure case 1 — Hallucinated implementation detail (naive LLM)

Question: "What header should I send with authenticated requests?"
What the system did: Naive LLM mode returned a confident answer naming a specific header (e.g. Authorization: Bearer) with a plausible description — but that exact phrasing did not appear in the docs.
What should have happened: Either return the actual documented header from AUTH.md, or refuse if the docs don't specify it. Confidently stating an unverified detail is worse than saying "I don't know."
Failure case 2 — Below-threshold refusal on a valid question (retrieval + RAG)

Question: "What database does this project use?"
What the system did: The word "database" appeared in several short headings that were filtered out (< 40 chars), and the scored chunks only mentioned it once each, falling below MIN_SCORE_THRESHOLD = 2. The system returned a refusal.
What should have happened: The answer was present in DATABASE.md. The failure was caused by the minimum chunk length filter discarding relevant headings. A lower threshold or a different chunking strategy would have caught this.
When should DocuBot say "I don't know based on the docs I have"?

When no chunks in the index contain any query word — the topic is simply not covered
When chunks do match but none score above the confidence threshold — weak, incidental overlap is not reliable evidence
When the LLM's system prompt is followed and the retrieved snippets genuinely don't answer the question
What guardrails did you implement?

MIN_SCORE_THRESHOLD = 2 in retrieve(): chunks with only 1 matching query word are excluded
Short chunk filter in _chunk_text(): paragraphs under 40 characters (headings, dividers) are discarded at index time
LLM system prompt constraints: the model is explicitly told not to use outside knowledge and to refuse rather than guess
Refusal message returned from answer_retrieval_only() and answer_rag() when retrieve() returns an empty list
6. Limitations and Future Improvements
Current limitations

Term-overlap scoring ignores word importance. "The" and "authentication" contribute equally to a score. A document about error handling will score 1 on any query containing the word "error," even if it's completely irrelevant.
Chunking is purely structural, not semantic. Splitting on \n\n means a chunk's boundaries are determined by how the author formatted the document, not by whether a paragraph is a coherent unit of meaning.
Single-word queries always refuse. With MIN_SCORE_THRESHOLD = 2, a precise one-word query like "pagination" will never return a result, even if a chunk is entirely about pagination.
No handling of synonyms or related terms. A query for "sign in" will not match a chunk about "authentication" or "login" unless those exact words overlap.
The index is static. If docs are updated after startup, the index is stale until the process restarts.
Future improvements

TF-IDF weighting: Downweight tokens that appear in many chunks (common words) and upweight rare, specific ones. This would fix the "error" problem and improve ranking precision without requiring embeddings.
Semantic/embedding-based retrieval: Replace or supplement term overlap with cosine similarity on sentence embeddings. This would handle synonyms, paraphrasing, and conceptual matches — the biggest gap in the current system.
Chunk overlap / sliding window: Include a few lines of overlap between adjacent chunks so context at paragraph boundaries isn't silently lost.
7. Responsible Use
Where could this system cause real-world harm if used carelessly?

The primary risk is confident misinformation from naive LLM mode. A developer who uses DocuBot to look up an API endpoint, a security parameter, or an environment variable name — and gets a hallucinated answer that looks correct — could ship a bug or a security vulnerability. The harm is proportional to how much the user trusts the output without verifying it against the source.

A secondary risk is over-trusting refusals as confirmation of absence. If DocuBot says "I don't know," that means the retrieval system didn't find sufficient evidence — not that the answer doesn't exist in the docs. A user might incorrectly conclude a feature is undocumented.

Guidelines for developers using DocuBot safely

Never use naive LLM mode for factual lookups. Reserve it for exploratory questions where approximate answers are acceptable. Use RAG or retrieval-only for anything that will influence code.
Treat refusals as "not found," not "doesn't exist." Always check the source documentation directly before concluding that information is absent.
Audit retrieved chunks before trusting the RAG answer. The source filenames are returned for a reason — spot-check that the cited chunk actually supports the claim.
Keep docs up to date. DocuBot is only as accurate as its source files. Stale documentation produces stale (and confidently delivered) answers.
